## OAuth (added mid-v1, not originally planned)

Both claude.ai's and ChatGPT's web "custom connector" flows turned out to require a real OAuth 2.1 dance (discovery, dynamic client registration, authorization code + PKCE) — a static bearer token in a header, which is all Claude Code CLI needs, isn't accepted by either web client. Since "usable from web clients" was the actual original goal, this pulled OAuth into v1 scope rather than deferring it.

- **Library**: `OpenIddict` (`OpenIddict.AspNetCore` + `OpenIddict.EntityFrameworkCore`), not hand-rolled — this is security-critical protocol code (PKCE validation, token issuance/expiry, redirect URI validation), exactly where you reach for a mature dependency instead of writing it yourself.
- **Dynamic client registration is hand-rolled**: OpenIddict 7.5.0 has no built-in RFC 7591 endpoint (verified by inspecting the shipped assembly — no `SetClientRegistrationEndpointUris` or similar exists). `OAuthEndpoints.cs`'s `POST /register` is a minimal subset (accepts `redirect_uris` + `client_name`, creates a public PKCE-required OpenIddict application, returns `client_id`) — not a full RFC 7591 implementation (no client update/delete, no auth on the registration endpoint itself since there's no secret to gate it with).
- **Storage**: SQLite (not JSON) at `~/.config/llama-mcp/oauth.db` (via `Environment.SpecialFolder.ApplicationData`, resolves cross-platform to `~/.config` on Linux/macOS and `%APPDATA%` on Windows) — chosen over hand-rolling JSON persistence for OpenIddict's application/authorization/token stores because that would mean reimplementing several elaborate store interfaces ourselves on a security-sensitive path; SQLite is a single local file, not a "lock-in" dependency, EF Core's provider is already battle-tested. Schema created via `Database.EnsureCreatedAsync()`, not EF migrations — no need for a `Migrations/` folder on a single-tenant local tool.
- **Signing/encryption keys**: a persisted RSA key at `~/.config/llama-mcp/signing.key` (`AppData.LoadOrCreateSigningKey()`), not OpenIddict's `AddDevelopmentEncryptionCertificate()` helpers — those lean on OS certificate stores, which behave inconsistently across Linux/Windows. A plain persisted RSA key file works identically on both and is why tokens survive a server restart instead of every restart invalidating every issued token.
- **"Consent screen" is not a real login system**: `GET/POST /connect/authorize` renders a bare HTML form asking for the *same* static bearer token from `AuthOptions` as the approval credential. There's exactly one owner of this server, so a full user/password/session system would be pure overhead — the real security boundary is still "do you know the token," same as before OAuth existed here.
- **Dual auth on the MCP endpoint**: the middleware in `Program.cs` accepts *either* the static bearer token (unchanged path, keeps Claude Code CLI/`claude mcp add --header` working exactly as before) *or* a valid OpenIddict-issued access token (`OpenIddictValidationAspNetCoreDefaults.AuthenticationScheme`). `/connect/*`, `/register`, and both `/.well-known/*` paths are always public (listed explicitly in `publicPaths`) — they must be reachable without a token for the dance to bootstrap at all.
- **`DisableTransportSecurityRequirement()` is required, not optional**: OpenIddict defaults to rejecting non-HTTPS requests. In our real topology, TLS *always* terminates at the Cloudflare tunnel — this process's own Kestrel listener only ever sees plain HTTP, even for genuine remote/HTTPS traffic (the quick tunnel forwards to `http://localhost:PORT`). Standard for anything behind a reverse proxy; don't remove this thinking it's a leftover dev shortcut.
- **`app.UseForwardedHeaders(...)` is also required**: `cloudflared` sends `X-Forwarded-Proto: https` (confirmed by inspecting raw request headers), but nothing reads it without this middleware — every generated URL (issuer, endpoints, redirects) would otherwise be built with `http://` even for real HTTPS traffic. `KnownIPNetworks`/`KnownProxies` are cleared because Kestrel only ever hears from `cloudflared` on loopback; there's no untrusted hop where a spoofed header could matter. Must run before `UseAuthentication()` below.
- **Protected Resource Metadata** (`GET /.well-known/oauth-protected-resource`, RFC 9728) and the `WWW-Authenticate: Bearer resource_metadata="..."` header on 401s are what let an MCP client discover the auth server automatically instead of requiring the user to paste a client ID manually.
- **`resource` indicator (RFC 8707) needs two separate fixes, not one**: MCP clients send `resource=<this server's own base URL>` on both `/connect/authorize` and `/connect/token`. (1) `options.DisableResourceValidation()` in `Program.cs` lifts the server-wide "is this resource registered at all" check — can't use `RegisterResources()` instead since the quick tunnel's URL isn't known until after `cloudflared` starts, well after the app's already running. (2) That alone isn't enough: OpenIddict separately requires each *application* to hold an explicit `Permissions.Prefixes.Resource + "<url>"` permission (constant value confirmed via reflection: prefix is `"rsrc:"`, undocumented in the XML docs) — without it you get a different error (`ID2192`, not `ID2190`) even with resource validation disabled globally. `OAuthEndpoints.cs`'s dynamic registration handler grants this permission for the server's own base URL to every client it creates.
- **`registration_endpoint` must be injected into the discovery document manually**: OpenIddict's configuration endpoint has no idea `/register` exists (it's our own hand-rolled endpoint, not something OpenIddict manages) and won't advertise it by default. Without this, claude.ai can't find dynamic registration and falls back to asking the user for a manually-entered client ID. Fixed via an `AddEventHandler<HandleConfigurationRequestContext>` inline handler in `Program.cs` that adds `context.Metadata["registration_endpoint"]`.
- **`app.UseAuthentication()` is required, not optional, even though nothing here uses cookie/session auth**: OpenIddict Server registers itself as an ASP.NET Core authentication scheme, and its non-passthrough endpoints (the discovery document is the one that matters here) only run through the full middleware pipeline — including `UseForwardedHeaders` above — if this is present. Without it, the discovery endpoint still responded (200 OK, valid JSON), but every URL in it came out `http://` instead of `https://`, silently regardless of the forwarded-headers fix. This one cost real debugging time — if OAuth URLs start coming back as `http://` again, check this middleware is still registered before assuming the forwarded-headers config is wrong.

## Running locally (dev, no tunnel)

```
dotnet build
Auth__BearerToken=<token> dotnet run
```

`Auth:BearerToken` has no default and fails startup validation if unset (`ValidateOnStart`) — set it via the `Auth__BearerToken` env var, not `appsettings.json` (that file is tracked in git, don't put a real token in it). `Backend:BaseUrl` defaults to `http://localhost:1234` (LM Studio's default). Point an MCP client at `http://localhost:5181/` (or whatever `--urls` you pass) with header `Authorization: Bearer <token>` — or drive the OAuth dance (`POST /register` → `GET/POST /connect/authorize` with the same token → `POST /connect/token`) if you're testing the web-client path. OAuth state lives in `~/.config/llama-mcp/` (`oauth.db` + `signing.key`) — delete both to reset to a clean slate.

For the full run (publish + tunnel), use `start.sh`/`start.ps1` — see README.md.

## What this is

An MCP server that lets a remote MCP client (Claude Desktop, Claude Code, etc.) talk to a local LLM inference server — llama.cpp (`llama-server`) or LM Studio's OpenAI-compatible API — running on a workstation, **without exposing a public port** on that workstation.

## Why this exists (goal, use to judge scope)

For someone who already pays for a Claude subscription, wiring in local agents only makes sense if it actually saves Claude tokens: offload basic, repetitive, or delegable work (extraction, classification, simple drafts, document/image analysis, etc.) to smaller/cheaper/custom models running on your own hardware, instead of spending Sonnet/Opus tokens on it. If a task doesn't fit that bar — it needs real reasoning, or delegating it costs more setup than it saves — there's no reason not to just stay on Sonnet directly. Use this as the filter for any new tool or feature: does it let you delegate real, repeatable work away from paid tokens, or is it a nice-to-have that doesn't move that needle.

## Architecture

```
MCP client (remote)  --HTTP/SSE over tunnel-->  Cloudflare Tunnel  -->  MCP server (on workstation)  --localhost-->  llama.cpp / LM Studio (OpenAI-compatible API)
```

Key decisions (settled — do not re-litigate without asking):

- **Topology**: the MCP server runs *on the same workstation* as llama.cpp/LM Studio, talking to it over `localhost`. Remote MCP clients reach the MCP server through the tunnel, not the other way around.
- **Tunnel**: Cloudflare Tunnel (`cloudflared`) is the default ingress. No public port is opened on the workstation; no dynamic DNS setup is needed for users.
- **Transport**: MCP over HTTP/SSE (required for remote clients, as opposed to stdio which only works for local same-machine clients).
- **Auth**: the MCP server owns authentication directly, not delegated to the tunnel layer. Two accepted credentials on the MCP endpoint: the static API key / bearer token (original design, still what Claude Code CLI uses), or an OpenIddict-issued OAuth access token (added mid-v1 once web clients turned out to require it — see "OAuth" section below).
- **Backend target**: the OpenAI-compatible chat/completions API, since both `llama-server` and LM Studio expose it — the MCP server should not need backend-specific code paths for the two.

## v1 scope

- **MCP SDK**: use the official `ModelContextProtocol` / `ModelContextProtocol.AspNetCore` NuGet packages for the Streamable HTTP transport — don't hand-roll JSON-RPC framing. Currently pinned to `2.0.0-preview.1` (installed with `--prerelease`, no stable release exists yet) — expect breaking API changes on upgrade, re-verify the `AddMcpServer()/WithHttpTransport()/WithTools<T>()/MapMcp()` pattern in `Program.cs` against the changelog before bumping.
- **Tools exposed**: `chat` (proxies `/v1/chat/completions`), `list_models` (proxies `/v1/models`), `health` (v1.1, see below), and `submit_job`/`get_job_status`/`get_job_result`/`cancel_job` (v1.2, see below). Nothing else until a real need shows up.
  - `chat` input: an OpenAI-style `messages` array (`role`/`content`), not a single flattened prompt string — needed for system prompts and multi-turn.
  - `chat` generation params (`temperature`, `max_tokens`, `top_p`, ...) are optional passthrough fields; if omitted, the backend's own defaults apply. No fixed values imposed server-side.
  - `chat`'s `model` field is optional; if omitted, falls back to a default model name from server config (consistent with the single fixed backend for v1).
- **Streaming**: none. `chat` returns the full completion in one response. The MCP host consumes tool results as a whole anyway, so token-by-token streaming buys nothing here — revisit only if generation length starts hitting client timeouts.
- **HTTP client to the backend**: plain `HttpClient` + `System.Text.Json` over minimal DTOs. Not the OpenAI SDK — too much for two endpoints.
- **Process model**: runs as a plain foreground process for now (`dotnet run` / manual start). No systemd/Windows service packaging until this is more than a dev setup.
- **Backends**: exactly one fixed backend base URL from config. No multi-workstation/multi-model routing until a real need shows up.
- **Cloudflare Tunnel**: quick tunnel (`cloudflared tunnel --url http://localhost:PORT`) for v1 — no Cloudflare account or domain required, one command, ephemeral `trycloudflare.com` URL. Named tunnel (stable hostname) is a later upgrade, and it has a hard prerequisite a script can't remove: the user must already have a domain on Cloudflare-managed DNS.
- **Distribution**: self-contained single-file publish per OS (`dotnet publish -r linux-x64|win-x64 --self-contained -p:PublishSingleFile=true`) — the user runs one binary, no separate .NET runtime install required.
- **Launcher scripts**: `start.sh` (Linux/macOS) and `start.bat`/`start.ps1` (Windows), each starting `cloudflared` quick tunnel + the MCP server together and printing the resulting URL and bearer token to paste into the MCP client config. Goal: one command/double-click to go from nothing running to a pastable connection.

## v1.1: health tool + multimodal input

Both shipped together, picked first from the v1.1+ backlog below because they map directly onto the project's two core use cases (see "Why this exists"): saving tokens on ordinary delegable work (`health` makes that path cheap to diagnose), and running a specialized local model like Qwen-VL for work Sonnet/Opus can't be cheaply substituted for otherwise (multimodal `chat`).

- **`health` tool**: wraps the existing `list_models` call in a try/catch instead of adding a new backend endpoint call — reuses what's already there rather than growing the backend client. Returns `Reachable`, `BaseUrl`, `Models` on success or `Error` on failure, so a client can tell "backend down" from "bad request" before spending a `chat` call.
- **Multimodal `chat`**: images travel as an optional `imageUrls` list (http(s) URLs or `data:` base64 URIs) on each `ChatMessageDto`, not as a separate tool parameter — mirrors how OpenAI's own vision API attaches images to a message's content, and rides the existing `messages` array without changing the tool's shape.
- **Wire format**: the OpenAI-compatible API needs `content` to switch from a plain string to an array of `{type: "text"|"image_url", ...}` parts once any image is attached. `LlamaBackendClient.ChatAsync` only pays for that rebuild (via `JsonNode`) when a message actually has `ImageUrls` — the plain-text path is still the original untouched `PostAsJsonAsync(request, ct)` call, so the already-verified v1 `chat` behavior can't regress.
- **Verified end-to-end**: both tools confirmed against a real LM Studio instance, and through the full remote path (quick tunnel + OAuth custom connector on claude.ai web) — `health` correctly reported reachability/model list, and `chat` correctly attached and got a response using an image content part, over the live tunnel URL, not just against a local mock.

## v1.2: timeout config, thinking control, async batch jobs, progress heartbeat

Triggered by a real test session against LM Studio (two local connectors, `google/gemma-4-e4b` and `qwen-agentworld-35b-a3b-local`) that surfaced two `chat` failure modes and one deeper architectural gap tied directly to "why this exists": a Claude web user with ~100 documents to run through a local Qwen-VL can't be served well by a single synchronous `chat` call.

- **Configurable backend timeout**: `AddHttpClient<LlamaBackendClient>` had no explicit `Timeout`, so it silently used `HttpClient`'s 100s default — the actual root cause of the hard invocation failures seen in testing (large/reasoning models routinely take longer than that). Now `Backend:TimeoutSeconds` (default 600s). No Kestrel-side change was needed alongside it: `HttpServerTransportOptions.IdleTimeout` (2h default) is a session-idle timeout between requests, not a per-request cap — confirmed by reading the shipped SDK XML docs rather than assuming.
- **`enableThinking` passthrough on `chat`/`submit_job`**: reasoning models (Qwen3.x family) burn `maxTokens` on a hidden "thinking" chain before the final answer — a prompt-level `/no_think` was tried during testing and didn't work, because the real mechanism is the OpenAI-API-compatible `chat_template_kwargs.enable_thinking` field, not a prompt hack. Added as `bool? EnableThinking` → `ChatTemplateKwargsDto` on `ChatCompletionRequestDto`, exposed as an optional per-call tool parameter — no default forced server-side, consistent with `chat`'s other generation params. Required no changes to `LlamaBackendClient`: `BuildMultimodalPayload` already serializes the whole DTO via `JsonNode` before touching `messages`, so any new top-level field rides both the plain-text and multimodal request paths for free. Verified against four different LM Studio-hosted models in a live test session (`google/gemma-4-12b-qat`, `qwen-agentworld-35b-a3b-local`, `qwen/qwen3.5-9b`, a custom `qwen3.6-27b-mtp` quant): the flag showed **no reliable, measurable effect on any of them** — same minimum token thresholds and same behavior whether it was `true`, `false`, or omitted, in one case outright non-deterministic across identical repeated calls. Confirmed the field is sent correctly on the wire, so this is an LM Studio/chat-template limitation (or the open upstream Qwen/LM Studio reasoning bug flagged during testing), not a connector bug. Keep the parameter — harmless, forward-compatible if a backend ever honors it — but don't rely on it actually reducing thinking-token cost on LM Studio today.
- **Async batch jobs (`submit_job` / `get_job_status` / `get_job_result` / `cancel_job`)**: the real fix for the "100 documents" story, not just a bigger timeout. A single sync `chat` call can't survive a multi-hour batch without either burning Claude tokens on a client-side wait/poll loop or losing all progress if the tunnel drops. Jobs are persisted in a separate SQLite store (`~/.config/llama-mcp/jobs.db`, own `JobDbContext`) deliberately *not* merged into `oauth.db` — that file's documented reset instructions ("delete oauth.db + signing.key") would otherwise silently destroy job data too. `JobProcessor`, a singleton `BackgroundService`, processes items sequentially (concurrency=1, matching realistic single-GPU local hardware). `jobId` is the only handle a client needs and is not tied to any MCP session — a dropped tunnel/expired session and a fresh reconnect can resume polling the same job; never cache job state in a per-session dictionary, it would defeat the reason this exists.
  - **Claim pattern, not a lock**: every item dequeue does a conditional `UPDATE ... WHERE Status = 'Pending'` before processing; 0 rows affected means "already claimed/cancelled." This is what makes the hybrid dispatch — an in-memory `Channel<Guid>` for low-latency pickup of freshly submitted items, plus a DB sweep on startup that resets any `Running` item back to `Pending` and requeues every `Pending` item left over from a prior crash — safe without a separate lock or distributed-coordination mechanism.
  - **`DateTimeOffset` doesn't work as an EF Core Sqlite `ORDER BY` column**: hit this directly during testing — the startup requeue sweep's `.OrderBy(i => i.CreatedAt)` threw `System.NotSupportedException` at runtime on the first restart test, because the SQLite provider can't translate `ORDER BY` over `DateTimeOffset`. Fixed by using `DateTime` (UTC) instead on `Job`/`JobItem` timestamp columns. Any future timestamp column on `jobs.db` needs `DateTime.UtcNow`, not `DateTimeOffset.UtcNow` — not a style choice, the alternative throws at query time, not at compile time.
  - **Verified locally via raw MCP JSON-RPC calls**: submit → sequential processing → per-item failure isolation (one item's backend error doesn't kill the rest of the batch) → status/result polling with pagination and status filtering, and specifically the crash-survival path (kill the process mid-batch, restart, confirm every item still reaches a terminal state instead of getting stuck).
  - **Since then, verified for real** through a live claude.ai web session over the full tunnel/OAuth path: full-document translation completed successfully via `submit_job` (confirmed on two different models), a 20-image vision batch classified successfully, and concurrency=1 confirmed live — a second job submitted while the first was still `running` sat entirely `pending` until the first finished, exactly as designed. See "v1.2 real-world findings" below for what surfaced beyond basic functionality.
- **Progress heartbeat on `chat`**: an `IProgress<ProgressNotificationValue>` tool parameter, auto-bound by the SDK and excluded from the tool's JSON schema (confirmed via the shipped SDK XML docs, and via a live `tools/list` call showing no `progress` property on `chat`), reports a "still generating" heartbeat every 15s while a `chat` call is in flight. This mitigates perceived dead-air on a single moderately-long call; it does **not** reliably prevent a client-side hard timeout — MCP clients can time out independently of server-side progress notifications (a real instance of this class of bug was found and cited during design, anthropics/claude-code#58687) — so it is not a substitute for the batch job system above on anything that could run long.

## v1.2 real-world findings (from live testing, not yet actioned)

A live multi-model test session through the tunnel/OAuth path (claude.ai web, four LM Studio models: `google/gemma-4-12b-qat`, `qwen-agentworld-35b-a3b-local`, `qwen/qwen3.5-9b`, a custom `qwen3.6-27b-mtp` IQ2_XXS quant) surfaced findings worth keeping for the next work session — not fixed yet, recorded here so they aren't lost:

- **New silent-failure pattern found, not yet fixed** *(actioned in v1.3, see above)*: on `qwen3.6-27b-mtp`, 6 of 20 vision-batch items returned empty `content` with `finishReason: "stop"` — not `"length"` like every previously-seen empty-response case. `"stop"` normally signals a clean, complete generation, so a caller checking only `finishReason` reads this as success. Confirmed via server-side token counts these items weren't hitting `maxTokens` (stopped spontaneously ~1500 tokens in, well under the 6000 budget) — a genuine model/generation anomaly, not a connector bug; the connector faithfully reported what the backend returned. **Concrete fix candidate for next session**: `chat`/`submit_job` results should flag empty `content` explicitly regardless of `finishReason` (e.g. an `IsEmpty`/`Warning` field, or a distinct job-item status instead of `Completed`) so a caller doesn't have to independently remember to check content-emptiness on every call.
- **Context-overflow handling is a backend config choice, not a connector concern, but worth documenting for users**: LM Studio's per-model context-overflow policy (sliding window / "truncate middle") can silently drop parts of the prompt — system instructions in particular — once total context is tight, producing a superficially valid but semantically broken response. Observed as heavy untranslated blocks on `qwen-agentworld-35b` and plausibly contributing to the empty-content pattern above on `qwen3.6-27b-mtp`. Recommend configuring LM Studio to error explicitly on context overflow instead of truncating silently, for any model orchestrated through this connector — a line for README/a future usage skill, not a code change here.
- **Concurrency=1 confirmed live, as designed** (see bullet above) — matches the single-GPU/single-model-instance assumption already documented, not a bug.
- **Model choice matters more than size or reasoning claims**: the smallest model tested (`qwen/qwen3.5-9b`) beat two larger ones on translation speed/quality and vision accuracy; the largest reasoning model (`qwen-agentworld-35b`) left large blocks untranslated and doesn't support vision at all. Nothing to build from this directly, but it's why the "never force-switch a model the user already loaded" constraint on the deferred `load_model`/`unload_model` tools below matters — the user has the empirical read on which local model is actually good for what, not the connector.
- **Enterprise router/proxy + model catalog** — expands the "Multi-backend / multi-model routing" deferred item below with concrete requirements surfaced during this session; see there.
- **A consumer-facing usage skill** *(written in v1.3, see `docs/consumer-guide.md`)* is worth writing separately from the connector itself: how to call this MCP effectively — size `maxTokens` generously (roughly 50-90% of the real context, never unbounded), verify vision capability with one image before a batch, treat empty `content` as failure regardless of `finishReason`, expect concurrency=1 (async removes blocking, not wait time), don't assume `enableThinking` does anything on LM Studio. This is guidance for whoever calls these tools (Claude, or another orchestrator), not connector code — not started.

## v1.3: empty-content detection, job timing surfaced, consumer guide

Directly actions the two concrete findings recorded above under "v1.2 real-world findings" (empty-content-with-`stop` and the timing/patience gap from `qwen3.6-27b-mtp`'s slow translation, invocation-test-results.md §13.3/13.4), plus the deferred consumer-facing usage skill.

- **Shared `IsEmptyContent` helper** (`ContentValidation.cs`): `string.IsNullOrWhiteSpace` in one place, routed through by both the `chat` and job paths — not duplicated per-caller. A backend can return `finishReason: "stop"` (a normal-looking, complete generation) with genuinely empty content; `"stop"` alone was never a safe success signal, confirmed directly in testing (6/20 items in one vision batch on `qwen3.6-27b-mtp`, well under its `maxTokens` budget, so not a length cutoff either).
- **`chat`**: adds `IsEmpty` (bool) to `ChatToolResult`, set from the shared helper. `chat` already returns content directly, so a flag on the result is the whole fix — no new tool, no new call shape.
- **Jobs**: adds a new terminal `JobItemStatus.CompletedEmpty`, set by `JobProcessor` instead of `Completed` when the backend call succeeds but content is empty. Additive to the `HasConversion<string>()` enum-as-string column — existing `Completed`/`Failed`/etc. rows are untouched, no migration needed. `get_job_status` gained a `CompletedEmpty` count (included in `Total`; doesn't affect `IsComplete`, since it's terminal); `get_job_result` needed no new field — `CompletedEmpty` items already surface through the existing per-item `Status` and `statusFilter`.
- **Job timing surfaced, not invented**: `JobItem.StartedAt`/`CompletedAt` were already persisted but never exposed. `get_job_status` now returns `RunningForSeconds` (elapsed since the currently-running item's `StartedAt`; concurrency=1 means at most one such item, null when none is running) — a small extra query on top of the existing status-count query, not a schema change. `get_job_result` now returns raw `StartedAt`/`CompletedAt` plus a convenience `DurationSeconds` (computed after fetching, not in the EF query, to avoid relying on the SQLite provider translating `TimeSpan` arithmetic). **Deliberately no auto-timeout or auto-cancel**: §13.3 showed a slow item (extreme quantization + heavy internal reasoning + low tok/s) can look stuck for a very long time and still complete successfully with good output — the server can't tell a genuinely slow item from a degenerate one, so it exposes the signal and leaves the cancel decision to the caller (`cancel_job` already exists for that).
- **`docs/consumer-guide.md`**: a short practical guide for whoever *calls* these tools (Claude or another orchestrator), distilled from invocation-test-results.md §9/§11.3/§13. Covers `maxTokens` sizing, treating empty content as failure, verifying vision capability before a batch, concurrency=1 expectations, the submit/poll/fetch + label pattern, and not relying on `enableThinking`. This is the deferred "consumer-facing usage skill" item from the v1.2 findings above — now written.
- **Unit tests (`tests/LlamaMcp.Tests/`, xUnit)**: deliberately narrow — cover only the pure, backend-independent logic that a live run wouldn't pin cheaply, i.e. `ContentValidation.IsEmptyContent`'s null/whitespace handling (the one branch both the `chat` `IsEmpty` and job `CompletedEmpty` decisions route through). The full behavior is still verified live against a real LM Studio instance (as with v1/v1.1/v1.2) — the unit tests complement that, they don't replace it. The web project's `Sdk.Web` glob compiles `**/*.cs` recursively, so `LlamaMcp.csproj` has a `Compile Remove="tests/**/*.cs"` to keep the test sources out of the main assembly. No solution file: build the app with `dotnet build`/`dotnet run` (resolves to the single root `.csproj`) and run tests with `dotnet test tests/LlamaMcp.Tests` (its project reference builds the app too).
- **Verified live** against a real LM Studio instance (gemma-4-e2b + qwen/qwen3.5-9b) over raw MCP JSON-RPC: `chat` `IsEmpty` both true (empty completion) and false (real content); job items reaching both `CompletedEmpty` and `Completed`; `RunningForSeconds` incrementing while an item runs and absent (null, omitted from JSON) when none does; `StartedAt`/`CompletedAt`/`DurationSeconds` populated on terminal items; `statusFilter` and the `CompletedEmpty` count in `get_job_status`. Not yet exercised over the full tunnel/OAuth path from claude.ai web (verified against the local backend directly), and no live vision batch in this round — those are the remaining gaps before calling it as battle-tested as v1.2.

## v1.4: token usage, better errors, cancel a running item, model stats, security hygiene, consumer skill

Triggered by a mix of already-collected-but-discarded evidence (usage/error data the backend was sending all along), a real live-testing pain point (a stuck item monopolizing the concurrency=1 queue with no lever but a server restart), and a security hygiene pass on code that had accreted a few rough edges since v1.

- **Token usage surfaced** (`ChatUsageDto` on `ChatCompletionResponseDto`, `PromptTokens`/`CompletionTokens` on `ChatToolResult` and `JobItem`/`JobItemResultDto`): the OpenAI-compatible `usage` object was already in every response and simply discarded. Lets a caller size `maxTokens` on a future call from real observed completion length instead of guessing. `jobs.db`'s `JobItems` table needed two new nullable INTEGER columns — see the schema-patch note below, this isn't a free schema change like `CompletedEmpty` was in v1.3 (that was additive to a string-converted enum column, not a new column).
- **jobs.db schema patch on startup, not EF migrations**: `Database.EnsureCreatedAsync()` only creates a database from scratch — it does not alter an existing table, so a user upgrading from a pre-v1.4 build would hit "no such column" the first time `PromptTokens`/`CompletionTokens` were touched. `Program.cs`'s `PatchJobItemsColumnsAsync` (ponytail-flagged) runs after `EnsureCreatedAsync`: it reads `PRAGMA table_info(JobItems)` and issues `ALTER TABLE JobItems ADD COLUMN ... NULL` only for columns actually missing. Deliberately generic over a small hardcoded list so the next additive column is a one-line addition, not a new migration story — still just for a single-user local db; if `jobs.db` ever needed real relational migrations (column drops/renames/type changes), that's the trigger to bring in EF migrations for real, not extend this patch further.
- **Backend error bodies surfaced**: `LlamaBackendClient`'s two `response.EnsureSuccessStatusCode()` calls threw with only the status code, discarding the body — a job item's `Error` became an opaque `"400 (Bad Request)"` with no way to tell "backend down" from "this model doesn't support images" without re-running the call by hand outside the connector (this literally cost a live diagnosis round). Both call sites now go through a shared `EnsureSuccessOrThrowAsync` that reads and truncates (~500 chars) the response body into the thrown `HttpRequestException`'s message.
- **`cancel_job` aborts an in-flight `Running` item, not just `Pending` ones**: previously the backend call ran to completion regardless of `cancel_job` — the only way to get unstuck from one glacially slow item monopolizing the concurrency=1 queue was restarting the whole server, a real pain live. A singleton `JobCancellationRegistry` (`ConcurrentDictionary<Guid, CancellationTokenSource>`, mirrors the existing job-channel DI pattern) holds a per-item CTS while `JobProcessor` awaits the backend call, linked to the outer `stoppingToken`. `JobTools.CancelJob` looks up any `Running` item for the job and cancels its registered CTS if present. The tricky part is distinguishing *why* the linked token fired inside `JobProcessor`'s catch block: if the outer `stoppingToken` was the cause (server shutdown), rethrow and leave the item `Running` — the existing startup requeue sweep (v1.2) picks it back up on next launch; only when it's specifically the per-item token (a real `cancel_job`) does the item get marked terminal, and as `Cancelled`, not `Failed`.
- **`get_model_stats`**: the single-owner-scale answer to "which local model is actually good for this task" — an empirical per-model profile (item counts by terminal status, average duration, average completion tokens) aggregated from `jobs.db`'s own history, joined `JobItem`→`Job` for the model name. SQLite/EF can't `AVG` a `TimeSpan` and there's no clean server-side median, so the terminal-item columns needed are pulled into memory and aggregated with LINQ (ponytail-flagged: fine at single-user job-history scale, revisit only if history grows huge). This is *not* the full "model catalog/ESB" from the v1.2 deferred backlog (see below) — no continuous refresh, no task-fit routing, no tok/s — just what's cheaply available from data the server already has, sized to how many models one user actually runs.
- **Security hygiene ride-along** (all cheap, stdlib, no new dependencies):
  - Constant-time token compare (`TokenComparer.TokensEqual`, `BackendOptions.cs`) via `CryptographicOperations.FixedTimeEquals`, replacing plain `==` on the bearer token in both the MCP auth middleware (`Program.cs`) and the OAuth consent-screen token check (`OAuthEndpoints.cs`) — a naive string comparison on a secret leaks timing information proportional to the first mismatched byte.
  - `signing.key` is now written with owner-only permissions (`UserRead | UserWrite` via `File.SetUnixFileMode`, guarded by `!OperatingSystem.IsWindows()`) instead of the default 644 — it's the RSA key that signs/encrypts every OAuth token this server issues.
  - OAuth dynamic client registration (`/register`) now has a hardcoded cap (50 applications) — the endpoint is intentionally unauthenticated (see the OAuth section above), so nothing previously stopped reconnect churn from creating an application per attempt. The cap is ponytail-flagged as a bound, not a fix: the real fix (purging tokenless clients left over from abandoned dances) is deferred until it's actually needed.
- **Consumer guide repackaged as an installable Claude skill** (`skills/llama-mcp-consumer/SKILL.md`): `docs/consumer-guide.md` (v1.3) was a doc someone had to remember to paste into context. As a skill with trigger-oriented frontmatter, an orchestrating Claude session picks up the usage lessons automatically when it's about to delegate work to llama-mcp, instead of relying on that. `docs/consumer-guide.md` is now a short pointer only — the skill is the single source of truth, kept from drifting apart by not duplicating the content. Expanded over the v1.3 doc with a keep-local-vs-escalate-to-Claude rubric (protects the "save paid tokens" north star from drifting into "do everything locally regardless of quality"), a sample-verify-don't-full-verify rule (grounded in a real test finding: a vision model answered a solid white image as "black" under a clean `finishReason: "stop"` — the kind of confidently-wrong output mechanical gates like `IsEmpty` can't catch, and full Claude verification of every item would erase the token savings the connector exists for), a chunking recommendation for long documents, and how to use the new v1.4 signals.

## v1.5: `chat` returns backend errors instead of throwing

A v1.4 QA finding: when the backend call in `chat` failed, the exception propagated out of the tool method and the MCP SDK masked it behind a generic "An error occurred invoking 'chat'." — the detailed backend error body v1.4 worked to surface (`EnsureSuccessOrThrowAsync`'s truncated response body) never reached the caller on the synchronous `chat` path, even though it already reached the caller on the job path (`item.Error`) and the server logs. `health` already established the right pattern for this connector — return failure as data, don't throw — so `chat` now mirrors it.

- **`ChatToolResult.Error`**: `string?`, null on success, the exception message on failure — same shape as `HealthToolResult.Error`.
- **`Chat` catches the backend call, doesn't let it escape**: on failure, returns `new ChatToolResult { Error = ex.Message, IsEmpty = true }` instead of throwing. The existing progress-heartbeat cleanup (`heartbeatCts.Cancel()` + `await heartbeatTask`) moved into a `finally` so it still runs on the error path, not just success.
- **Genuine client cancellation still propagates**: `catch (OperationCanceledException) when (cancellationToken.IsCancellationRequested) { throw; }` runs before the general `catch (Exception ex)`, same discrimination `JobProcessor` already uses for its per-item cancellation token — a backend-timeout `TaskCanceledException` (whose cancellation didn't come from the caller's token) is reported as an `Error` result, but an actual client cancel is rethrown, not swallowed.

## Deferred to v1.1+ (not decided, not started)

Explicitly out of scope for v1, kept here so they aren't forgotten or re-debated from scratch:

- **Streaming** chat responses (token-by-token), if generation length ever hits client timeouts.
- **systemd / Windows service** packaging, once this is more than a dev setup.
- **Multi-backend / multi-model routing** (more than one workstation or model to pick from). Real requirements surfaced during v1.2 testing (see "v1.2 real-world findings" above), two distinct concerns rather than one feature: a **router/proxy** (route by task fit, not just availability; centralize queue/concurrency across backends since each individual backend processes one item at a time; enforce policies like `maxTokens` sizing uniformly instead of every consumer rediscovering them) — still fully deferred, not started — and a **model catalog** (an empirical per-model profile — real tok/s, observed silent-failure rate, capability actually verified rather than declared — used to pick the best-fit model for a task, refreshed continuously rather than assessed once) — **partially addressed at single-owner scale by `get_model_stats` (v1.4)**: it gives the empirical-profile data itself (item outcomes, duration, completion tokens from real job history), but none of the catalog's other properties — no continuous background refresh (only reflects jobs already run), no task-fit-aware routing, no multi-backend awareness, no tok/s. The full ESB-style catalog and the central router both remain deferred; `get_model_stats` is a cheap data source a future router could build on, not a replacement for one.
- **Named Cloudflare Tunnel** (stable hostname), once a user has a domain on Cloudflare DNS.
- **`embeddings` tool** (proxy `/v1/embeddings`), if a RAG-style use case shows up.
- **`load_model` / `unload_model` tools**, if backend model switching becomes a real need — note this would likely reintroduce backend-specific code paths (LM Studio vs `llama-server` differ here). Constraint already decided for whenever this is built: **never force-switch a model that's already loaded** — if LM Studio already has a model loaded when a tool call comes in, use it as-is (check via `list_models`/`health` first); only select/load a specific model when nothing is currently loaded. The user manages what's loaded locally (e.g. picking a vision-capable model like `google/gemma-4-12b-qat` for an image task) and that choice shouldn't be silently overridden by the MCP server.
- **Official MCP Registry publishing** — deferred, not just unstarted. `server.json` requires either a published package (`npm`/`pypi`/`nuget`/`oci`/`mcpb`) or a fixed public `remotes` URL; llama-mcp has neither (self-contained binary, no package registry; ephemeral per-user Cloudflare quick tunnel, no stable URL). Closest fit is publishing a NuGet global-tool package purely to satisfy the registry's verification requirement — revisit if that packaging is ever done for its own sake, don't build it just to unlock a registry listing.
